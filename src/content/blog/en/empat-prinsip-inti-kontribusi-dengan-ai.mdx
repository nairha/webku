If you know me well, you've probably heard some version of this before. But I want to write it down clearly so it's accessible to anyone considering contributing to projects I maintain.

I'm not against AI. Quite the opposite — I follow its development closely and think LLMs are among the most useful tools ever created for text and code work. But there's a way of using them that I consider sound, and a way that I consider problematic — especially in an open source context.

## Communication Should Come From You, Not From the Model

The first principle, and the one I emphasize most: don't let an LLM speak for you. When you send an email, write a comment on an issue, or explain something in a pull request — I want to read your voice, not text that's been polished by a model.

I don't care about grammatical perfection. I don't care if the sentence is rough or there's a typo or two. What I care about is whether there's a person behind those words who is actually thinking and trying to communicate something. Open source contribution, to me, is about connection between people — not about who can produce the most polished-sounding text.

## Understand What You're Contributing

Using an LLM to help write or review code is normal and perfectly fine. But before you submit anything to an open source project — whatever it is — you need to genuinely understand what you're submitting.

This isn't a formality. If there's a bug in code you contributed and someone asks about it, you should be able to explain it. If there's a design decision that needs discussion, you should have an opinion you actually believe in, not just one the model recommended. If you can't explain your own code, the contribution adds more burden than it resolves.

## Vibe Coding Doesn't Grow Anyone

I strongly support using LLMs to speed up work. But there's a significant difference between using AI as a tool and handing the entire direction of a project over to the model. The latter is often called vibe coding — sit down, type a prompt, accept the output, move on.

The problem isn't that the output is necessarily bad. The problem is what happens to you in the process. If you're never involved in technical decisions, never wrestling with problems that don't have clean answers in training data, you don't grow. And an open source project that depends on contributors who aren't growing is a project that's slowly stagnating.

## Transparency Is About Integrity

The fourth principle I hold is honesty about AI use. If you used an LLM to help with research, write a draft, or generate part of the code — say so. Not because it's shameful, but because it's honest.

In many professional contexts, disclosing AI use is a form of integrity. It helps people reviewing or reading your work understand the context more fully. And in open source communities built on trust, that kind of honesty carries more weight than it might appear.
